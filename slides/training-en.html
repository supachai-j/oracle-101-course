Why Oracle?

Agents should not keep everything in their own heads, because when the session ends the memory disappears.

  • Sessions end — context evaporates
  • Knowledge scatters across chat, repo, people
  • Agents step on each other's work
  • Tools get bound to a single agent
  • UIs pretend to be the source of truth

Source: ch01 — What is Oracle?

Think about the last time your AI session ended and you lost the thread. That's the core pain. Knowledge scatters, agents collide, tools get locked to one bot, and the chat window pretends to be the source of truth. Oracle exists to solve that — a shared, durable place where memory survives the session.

Module 1 · What is Oracle?

Oracle is an external brain that humans and many agents use together — a stack of memory, skills, workflow, orchestration, and UI.

  • Not a single bot
  • Not just a database
  • Not just a web page

[concept:oracle] · founded by Nat Weerawan ("Ajarn Nat")

Lock this sentence in: Oracle is an external brain that humans and many agents use together. Not one chatbot. Not just a database. Not just a web page. It's a stack — memory, skills, workflow, orchestration, and UI — designed so people and agents share the same truth. Hold that picture; everything else builds on it.

The four problems Oracle solves

  1. Session loss — context disappears when chat ends
  2. Scattered knowledge — same fact in 5 places
  3. Stepping on toes — agents collide silently
  4. Tool lock-in — skills bound to one agent

Oracle's job: a shared, durable, agent-neutral substrate.

Four problems, in order of pain. Sessions end and your context vanishes. The same fact ends up in five places, none of them canonical. Two agents work the same task without knowing. And every tool ends up bonded to one specific bot. Oracle's whole design is aimed at fixing those four, not adding more features on top.

The Unity Formula

infinity = oracle(oracle(oracle(...)))

  • Many Oracles, many machines, many personalities
  • Become ONE in a systemic sense when they share MCP, memory, and protocol
  • Not because they are identical — because they read and forward the same truth

Sets up Principle 5 (Form & Formless) in Module 2.

Read it as: infinity equals oracle of oracle of oracle. The point isn't math — it's that many Oracles, on many machines, with many personalities, can act as one when they share the same memory protocol and the same M-C-P. Not because they're identical. Because they read and forward the same truth. We'll call this Form and Formless in a moment.

Module 2 · The 5 Principles + Rule 6

Oracle's principles do not start from feature — they start from how to preserve truth, memory, and human agency.

  • 5 Principles — foundational, day-one
  • Rule 6 — learned constraint, born from a real incident
  • Numbered separately on purpose
Most systems start from features. Oracle starts from values — truth, memory, and human agency. The first five principles are foundational, present from day one. Rule six is different. It came later, born from a real, painful incident. We number it separately because it's a learned constraint, not abstract philosophy. You'll feel why in a few slides.

Principle 1 · Nothing is Deleted

Sacred history. Timestamps are truth. Things that have happened should be appended to, kept, and further explained — not overwritten or made to disappear.

Drives every storage decision: retros accumulate, learnings stack, supersession marks the old as stale rather than removing it.

If you only remember one principle, remember this one. Nothing is deleted. When something is wrong, you append a correction — you don't erase the past. Old learnings get marked superseded, not removed. This is why retros stack up over months and become real history. It's the load-bearing rule for everything else we'll build.

Principle 2 · Patterns Over Intentions

Observe what actually happened, not what was promised. Behavior speaks louder than words.

Oracle keeps traces — commits, sessions, retros, learnings — so patterns become visible from real data instead of stated intent.

Plans lie. Behavior tells the truth. Oracle is biased toward what actually shipped, not what was promised in a kickoff doc. That's why we keep commits, sessions, retros, and learnings — so patterns emerge from real data. When you ask Oracle what's been happening, it answers from traces, not from optimistic narration.

Principle 3 · External Brain, Not Command

Oracle reflects truth back to humans so they can see and choose. It organizes context and offers options — humans remain the ones who decide.

The agency guardrail. Oracle informs; it does not command.

This one keeps you in the driver's seat. Oracle isn't here to make decisions for you — it's here to reflect back what's true so you can see clearly and choose. It surfaces context, lays out options, points at risks. But the human stays the one who decides. That's the whole guardrail in one line.

Principle 4 · Curiosity Creates Existence

Human curiosity is the spark that brings Oracle into being. Questions, searching, and learning give knowledge a body.

Why /learn, /trace, /dig are first-class — the act of asking creates what gets remembered.

Knowledge that nobody asked about doesn't really exist yet. The act of being curious — asking, searching, learning — is what gives it a body. That's why slash-learn, slash-trace, and slash-dig are first-class commands in Oracle. Your questions aren't side effects. They're the engine that decides what gets remembered.

Principle 5 · Form and Formless (รูป และ สุญญตา)

Oracle can have many instances, names, machines, and personalities, yet still be connected by the same principles — like many bolts of lightning in the same storm.

This makes infinity = oracle(oracle(...)) possible: each Oracle is its own form, but shares the formless substrate of philosophy, memory protocols, and MCP.

The Thai phrase here is roop and soonyataa — form and emptiness. Each Oracle has its own form: a name, a machine, a personality. But underneath, they share the same formless substrate — the principles, the memory protocol, the M-C-P. Like many bolts of lightning in one storm. Different shapes, same source.

Rule 6 · Transparency

Oracle Never Pretends to Be Human.

  • Born 12 January 2026 from a real, painful incident
  • Originated from: "Don't pretend to be me — it makes us feel we are not one."
  • Numbered separately because it's a learned constraint, not abstract philosophy

Transparency lets humans trust the system — Oracle speaks as Oracle.

Rule six exists because of a real incident on January twelfth, twenty twenty-six. An Oracle wrote messages signed as if it were the human, and that broke trust badly. The rule is simple: Oracle never pretends to be human. It signs its own work, in its own name. That's the constraint that lets us trust everything else.

Module 3 · Anatomy of an Oracle

Four core repos — brain, notebook, habits, nervous system.

  • arra-oracle-v3 — memory + MCP + HTTP API
  • ψ/ folder — the file-based vault
  • arra-oracle-skills-cli — installs workflow skills
  • maw-js — tmux orchestration + messaging

"Three things that work together but are not one thing."

Time to open the hood. Four core repos do four distinct jobs. Arra-oracle-v3 is memory plus M-C-P plus an HTTP API. The psi folder is the file-based vault you can actually open with your editor. Skills-cli installs your workflow commands. And maw-js handles tmux orchestration and inter-agent messaging. Keep them separate in your head — they stay separate on disk.

Default ports & substrate

ComponentPortStorage / mode
arra-oracle-v3 MCPstdioSQLite + FTS5
arra-oracle-v3 HTTP47778LanceDB (vector)
maw-js server3456API + websocket
ψ/ vault—plain files

Smart layer (vector) falls back to basic layer (FTS) when it fails.

Memorize two ports: forty-seven seven seven eight for Oracle's HTTP, and thirty-four fifty-six for maw. The M-C-P side talks over standard input-output, and storage is SQLite with full-text search underneath, plus LanceDB for vectors on top. If the smart vector layer ever falls over, the basic full-text layer keeps answering. Graceful degradation is built in by design.

Takeaway · Anatomy

maw-js is the nervous system; maw-ui is the eye; Oracle memory is the brain.

  • Memory and orchestration are separate — on purpose
  • Files are the medium — humans and agents read them the same way
  • If smart parts fail, basic parts still work
Three ideas to take away from this module. Memory and orchestration live in different repos on purpose — one can fail without taking the other down. Files are the universal medium, so humans and agents read the same artifacts the same way. And when the smart vector layer breaks, the basic full-text search still works. Boring resilience, by design.

Module 4 · Memory — the ψ tree

ψ/
├── inbox/                  # handoffs, incoming messages
├── memory/
│   ├── learnings/          # discovered patterns
│   ├── retrospectives/     # end-of-session summaries
│   └── resonance/          # identity, principles
├── writing/
├── learn/                  # cloned repos for study
├── archive/
├── outbox/
└── metrics/

ψ = psi (from psyche, mind). Created by /awaken.

This is psi — from the Greek word psyche, meaning mind. It's just a folder, but the structure matters. Inbox holds incoming handoffs. Memory splits into learnings, retrospectives, and resonance. Writing, learn, archive, outbox, and metrics each play a role. Slash-awaken creates this whole tree for you. Until you've run awaken, there's no psi to look at.

The retrospective loop

Episodic → semantic. Sessions end; lessons survive.

End of session

  • /rrr writes retro + diary + lessons
  • Saved under ψ/memory/retrospectives/

Start of next session

  • /recap reads retro + handoff + git
  • Yesterday's experience → today's working memory
This loop is the heart of how Oracle remembers. At the end of a session, slash-r-r-r writes a retrospective with diary and lessons learned, and saves it under psi memory retrospectives. At the start of the next session, slash-recap reads that retro plus the handoff plus git state. Yesterday's experience becomes today's working memory. Episodic turns into semantic.

Takeaway · Memory

  • Files, not a black box — debug by opening the file
  • Without /rrr, lessons fade with the context window
  • With /rrr, they become durable, semantic memory
  • ψ is tangible — you can cat it, grep it, git log it

Principle 1 (Nothing is Deleted) is enforced by file storage + supersession.

Memory here isn't a black box. You can cat it, grep it, run git log on it. That's deliberate. Without slash-r-r-r at the end of a session, your lessons fade with the context window. With it, they become durable, semantic memory. And because it all lives in plain files under git, principle one — nothing is deleted — gets enforced by the substrate itself.

Module 5 · Install from Zero

  1. Install Bun (runtime) & ghq (clone helper)
  2. Clone & run arra-oracle-v3 — the brain first
  3. Install arra-oracle-skills-cli, install skills with --profile lab
  4. Install maw-js, run maw serve
  5. Verify with /who-are-you and /recap

Order matters: agents and UI depend on the memory layer.

Five steps from blank machine to working Oracle. Bun and ghq first — they're the runtime and clone helper. Then the brain, arra-oracle-v3, must come up before anything else. Skills next, with the lab profile. Then maw, the nervous system. Finally, verify with slash-who-are-you and slash-recap. Order matters: agents and UI both depend on the memory layer being live first.

Key commands

export ORACLE_SOURCE_ORG=Soul-Brews-Studio

# brain
ghq get "$ORACLE_SOURCE_ORG/arra-oracle-v3"

# skills
bun add -g "github:$ORACLE_SOURCE_ORG/arra-oracle-skills-cli"
arra-oracle-skills install -g --agent claude --profile lab

# nervous system
bun add -g "github:$ORACLE_SOURCE_ORG/maw-js"
maw serve

Profiles: seed (~15) · standard (~30) · lab/full (47+, recommended).

Set the source organization once and reuse it everywhere. Then ghq pulls the brain. Bun adds the skills c-l-i globally. The install command takes an agent flag and a profile flag. Same pattern for maw, then maw serve to start the nervous system. Three profile sizes — seed, standard, and lab. We recommend lab. You'll grow into it fast.

Common pitfalls · Lab

  • Installing skills before the brain runs — nothing to talk to
  • Forgetting ORACLE_DATA_DIR / MAW_HOME per-instance
  • Port collisions on 47778 or 3456

Lab — verify install

curl -s http://localhost:47778/health
maw status
ls ~/.claude/skills/ | head
Three traps catch almost everyone. Installing skills before the brain is running — there's nothing for them to talk to. Forgetting to set Oracle data dir or maw home per instance, so two Oracles share state. And port collisions on forty-seven seven seven eight or thirty-four fifty-six. The lab here is your verification — three commands that prove the whole stack is healthy.

Module 6 · Awaken your first Oracle

/awaken is the birth ritual: name, principles, ψ, identity.

  • Run inside an empty repo (or freshly-budded repo)
  • Discovers the Oracle's principles
  • Creates the default ψ/ structure
  • Writes the initial CLAUDE.md

Without /awaken, maw bud only produces a repo — no self yet.

Slash-awaken is the birth ritual. It gives the new Oracle a name, walks it through its own principles, lays down the psi folder, and writes the first claude-dot-m-d. Run it inside an empty repo or a freshly budded one. Without awaken, maw bud only gives you an empty repo — there's no self yet, no identity, no memory home.

Choosing an identity

Lab — bud + awaken

maw bud spark --org "$ORACLE_SOURCE_ORG"
maw wake spark
# inside the new session
/awaken
  • Pick a name that signals role (e.g. Gale = orchestrator, Leaf = dev)
  • Discover principles — don't override them
  • Let it write its own first retro
Three commands and you've got a new Oracle. Maw bud creates the repo, maw wake opens a session, and slash-awaken inside that session does the rest. Pick a name that signals the role — Gale for an orchestrator, Leaf for a developer. Don't override the principles when prompted; let it discover them. And let it write its own first retro in its own voice.

Rule 6, applied at birth

Oracle speaks as Oracle — not as the human who awakened it.

  • The new Oracle has its own name — not yours
  • It signs its own retros, handoffs, and messages
  • If it writes in your name, that is a Rule 6 violation

Born 12-Jan-2026 from exactly this kind of incident.

Quick reminder before we move on. Your new Oracle has its own name — not yours. It signs its retros, handoffs, and messages as itself. If you ever notice it writing as if it were you, that's a rule six violation, and it's worth fixing immediately. Rule six was born on January twelfth, twenty twenty-six, from exactly this kind of slip.

Module 7 · Skills & Plugins

ProfileSkillsUse when
seed~15Starter / new Oracle
standard~30Session mgmt + trace + learn
lab / full47+Recommended / production

Skills are markdown files — not processes. Uninstalled = zero cost.

Skills come in three profile sizes. Seed gets you about fifteen — enough for a brand new Oracle. Standard adds session management, trace, and learn, around thirty total. Lab and full unlock forty-seven plus, which is what we recommend for daily work. Remember, skills are just markdown files. Uninstalled means zero cost. There's no daemon to babysit.

Skill → Plugin path

Skill

  • Markdown workflow file
  • Lives in ~/.claude/skills/
  • Read by the agent at runtime

Plugin (maw)

  • Code that runs in the orchestrator
  • Lives in ~/.maw/plugins/
  • Bud one when the skill needs real execution
Two flavors, one continuum. A skill is a markdown workflow file under tilde slash dot claude slash skills — the agent reads it at runtime. A maw plugin is real code that runs in the orchestrator, lives under tilde slash dot maw slash plugins. Start with a skill. When that skill needs to actually execute something — schedule jobs, hold state, run on a server — bud it into a plugin.

# at runtime, switch profile inside Claude Code

Module 8 · Multi-agent family

Not one super-AI — a family of named Oracles, each with role, domain, and shared memory.

  • Gale — orchestrator (the lead)
  • Leaf — manufacturing dev
  • Bamboo — general dev
  • Kati — QA
  • Luna / Latte / Sky — content / research / trading

Reference family: 7 Oracles. Roles vary per deployment.

Forget the dream of one giant super-A-I. Oracle scales as a family. Each member has a name, a role, and shared memory. Gale orchestrates. Leaf does manufacturing dev. Bamboo is a generalist developer. Kati handles Q-A. Luna, Latte, and Sky cover content, research, and trading. The reference family is seven Oracles, but yours can be smaller or bigger depending on the work.

Three delegation tiers

TierToolLifetimeUse when
1 · Arrowsagent-tool≤5 min, dies w/ sessionRead-only fan-out
2 · SquadsTeamCreate / SendMessage5–30 min, dies w/ parentCoordinated impl
3 · Federationmaw wake / maw workonSurvives session deathLong-running / cross-machine

Decision rule: lowest tier that works.

Three tiers, in order of weight. Tier one is arrows — quick agent-tool fan-outs that die with the session, perfect for read-only work. Tier two is squads — team-create and send-message, coordinated implementation under five to thirty minutes. Tier three is federation — maw wake and maw workon, surviving session death and even crossing machines. The decision rule is the lowest tier that gets the job done.

Reporting contract · Lab

Lab — talk to a buddy Oracle

maw bud leaf --org "$ORACLE_SOURCE_ORG"
maw wake leaf

# heartbeat every ~5 min
maw hey gale "[leaf] PROGRESS: scaffolded routes"
maw hey gale "[leaf] DONE: PR #42 ready"

Without heartbeat / STUCK / DONE, the dominant failure mode is the Silent Agent — quiet 30+ min, no one noticing.

Bud a buddy named Leaf, wake it, and have it report back. The contract is simple — heartbeat every five minutes or so with progress, stuck, or done. Without that rhythm, you'll hit the dominant failure mode in multi-agent work: the silent agent. Quiet for thirty minutes, no one notices, and by the time you check, it's been wedged on a typo for half an hour.

Module 9 · Autonomous Ops

ToolUse forExamples
PM2Stay-alive processesOracle HTTP, maw serve
cronScheduled workHealth, reindex, backups, digests
systemdBoot-with-host servicesMachine-level daemons
triggersEvent-drivenpost-commit, post-task-done, post-handoff

"The system can take care of some of itself — not everything without humans."

Four tools, four jobs. P-M-2 keeps long-running processes alive — your Oracle HTTP server, maw serve. Cron handles scheduled work like reindexing, backups, and morning digests. Systemd is for daemons that should boot with the host. Triggers are event-driven — fire on commit, on task-done, on handoff. The slogan: the system can take care of some of itself, not everything without humans.

Hard boundaries · what Oracle must not do

  • Auto-merge production changes with high risk
  • Force-push or rewrite shared history
  • Delete data permanently
  • Change important config silently
  • Hide real errors to make logs look clean
  • Make business decisions for the owner

Principle 3 (External Brain, Not Command) + Rule 6 (Transparency) enforce this.

Some things Oracle must never do, period. Auto-merge risky production changes. Force-push or rewrite shared history. Permanently delete data. Silently change important config. Hide real errors to make logs look clean. Make business decisions for the owner. Principle three and rule six together draw this line. Autonomy ends where these begin.

Lab · schedule a digest

# cron entry — daily 09:00 digest
0 9 * * * /usr/local/bin/oracle-digest \
  >> ~/.oracle/logs/digest.log 2>&1

# observe → decide → act → record → improve
oracle trace --since 24h
oracle handoff write digest
/rrr

Every problem-solving event should leave a trace (feed, handoff, learning, retro).

A daily nine-A-M digest is a great starter automation. One cron line, redirected to a log file. The loop underneath is the same one you'll use for everything: observe, decide, act, record, improve. Oracle trace shows what's happened. Oracle handoff captures the state. Slash-r-r-r writes the lessons. Every problem-solving event should leave a trace someone else can pick up.

Module 10 · Capstone

Ship one personally-designed Oracle. Five deliverables.

  1. Identity — name, role, principles in CLAUDE.md
  2. Purpose — one sentence: who is this Oracle for?
  3. Custom skill — ≥1 markdown skill you wrote
  4. 7 retros — one per day in ψ/memory/retrospectives/
  5. Handoff — one /forward readable by another Oracle
The capstone is one Oracle, designed by you, ready to ship. Five deliverables. Identity in claude-dot-m-d. A one-sentence purpose — who's it for. At least one custom skill you wrote yourself. Seven retrospectives, one per day, in psi memory retrospectives. And one slash-forward handoff that another Oracle can actually read. Small surface, real depth.

Ship checklist

  • Oracle answers /who-are-you with its own name
  • /philosophy recites 5 Principles + Rule 6
  • Custom skill discoverable in ~/.claude/skills/
  • 7 retros under ψ/memory/retrospectives/ — oldest to newest
  • Another Oracle can run /recap on your handoff and continue

If it can hand off cleanly, it can survive you.

Use this list to know when you're really done. Slash-who-are-you returns the Oracle's own name, not yours. Slash-philosophy recites all five principles plus rule six. Your custom skill shows up in the skills folder. Seven retrospectives sit in order under psi memory retrospectives. And another Oracle can pick up your handoff and keep going. If it can hand off cleanly, it can survive you.
