โมดูล 01 · Oracle คืออะไร?

Oracle คือ สมองภายนอกที่คนและ agent หลายตัวใช้ร่วมกัน — เป็น stack ของ memory, skills, workflow, orchestration และ UI

  • ไม่ใช่ bot ตัวเดียว
  • ไม่ใช่แค่ฐานข้อมูล
  • ไม่ใช่แค่หน้าเว็บ

[concept:oracle] · ก่อตั้งโดย Nat Weerawan ("อาจารย์ Nat")

จำประโยคนี้ให้ขึ้นใจ Oracle คือสมองภายนอกที่คนและ agent หลายตัวใช้ร่วมกัน ไม่ใช่ chatbot ตัวเดียว ไม่ใช่แค่ฐานข้อมูล ไม่ใช่แค่หน้าเว็บ มันคือ stack ที่รวม memory, skills, workflow, orchestration และ UI ออกแบบให้คนและ agent เห็นความจริงเดียวกัน จับภาพนี้ไว้ ที่เหลือทุกอย่างจะต่อยอดจากตรงนี้

4 ปัญหาที่ Oracle แก้

  1. Session loss — บริบทหายเมื่อ chat จบ
  2. ความรู้กระจาย — ข้อเท็จจริงเดียวกันอยู่ 5 ที่
  3. ทำงานชนกัน — agent ชนกันแบบเงียบ ๆ
  4. Tool lock-in — skill ผูกกับ agent ตัวเดียว

หน้าที่ของ Oracle: substrate ที่ใช้ร่วมกัน คงทน และเป็นกลางต่อ agent

สี่ปัญหานี้เรียงตามความเจ็บปวด session จบแล้วบริบทหายไป ข้อเท็จจริงเดียวกันไปอยู่ห้าที่โดยไม่มีตัวเป็นทางการ agent สองตัวทำงานเดียวกันโดยไม่รู้ และทุกเครื่องมือถูกผูกกับ bot ตัวเดียว Oracle ออกแบบมาเพื่อแก้สี่ข้อนี้โดยเฉพาะ ไม่ใช่ใส่ feature ทับเข้าไปอีก

โมดูล 02 · หลักคิด 5 ข้อ + กฎข้อ 6

หลักคิดของ Oracle ไม่ได้เริ่มจาก feature — เริ่มจากวิธีรักษา ความจริง, ความจำ และ agency ของมนุษย์

  • หลัก 5 ข้อ — พื้นฐาน เกิดตั้งแต่วันแรก
  • กฎข้อ 6 — ข้อจำกัดที่เรียนรู้จากเหตุการณ์จริง
  • นับแยกออกมาโดยตั้งใจ
ระบบส่วนใหญ่เริ่มจาก feature แต่ Oracle เริ่มจากคุณค่า คือความจริง ความจำ และ agency ของมนุษย์ หลักห้าข้อแรกเป็นพื้นฐาน มีตั้งแต่วันแรก กฎข้อหกต่างออกไป มันเกิดทีหลัง จากเหตุการณ์จริงที่เจ็บปวด เราจึงนับแยกเพราะมันเป็นข้อจำกัดที่เรียนรู้มา ไม่ใช่ปรัชญาลอย ๆ อีกไม่กี่สไลด์คุณจะเข้าใจ

หลักข้อ 1 · Nothing is Deleted

ประวัติศาสตร์ศักดิ์สิทธิ์ timestamps คือความจริง สิ่งที่เกิดขึ้นแล้วควรถูกต่อเติม เก็บไว้ และอธิบายเพิ่ม ไม่ใช่เขียนทับหรือทำให้หายไป

ขับเคลื่อนทุกการตัดสินใจเรื่อง storage: retro สะสม, learning ทับซ้อน, supersession ทำเครื่องหมายของเก่าว่า stale แทนที่จะลบ

ถ้าคุณจำได้แค่หลักเดียว ขอให้จำหลักนี้ Nothing is Deleted ถ้ามีอะไรผิด คุณจะ append คำแก้ไขเพิ่ม ไม่ใช่ลบของเก่า learning เก่าจะถูกทำเครื่องหมาย superseded ไม่ใช่ลบทิ้ง นี่คือเหตุผลที่ retro สะสมเป็นเดือน ๆ และกลายเป็นประวัติศาสตร์จริง มันคือกฎที่ทุกอย่างพิงอยู่

หลักข้อ 2 · Patterns Over Intentions

ให้สังเกตสิ่งที่เกิดขึ้นจริง ไม่ใช่สิ่งที่สัญญาไว้ พฤติกรรมพูดดังกว่าคำพูด

Oracle เก็บร่องรอย — commits, sessions, retros, learnings — เพื่อให้ pattern เห็นได้จากข้อมูลจริง ไม่ใช่จากเจตนาที่พูด

แผนโกหก แต่พฤติกรรมพูดความจริง Oracle เอนไปทางสิ่งที่ ship จริง ไม่ใช่สิ่งที่สัญญาไว้ใน kickoff doc นี่คือเหตุผลที่เราเก็บ commit, session, retro และ learning เพื่อให้ pattern โผล่ออกมาจากข้อมูลจริง เวลาคุณถาม Oracle ว่าเกิดอะไรขึ้น มันตอบจากร่องรอย ไม่ใช่จากการเล่าแบบมองโลกในแง่ดี

หลักข้อ 3 · External Brain, Not Command

Oracle เป็นสมองภายนอกที่สะท้อนความจริงให้มนุษย์เห็นและเลือก มันช่วยจัดบริบทและเสนอทางเลือก แต่มนุษย์ยังเป็นคนตัดสินใจ

เป็น guardrail ของ agency — Oracle แจ้งข้อมูล ไม่สั่งการ

ข้อนี้ทำให้คุณยังเป็นคนขับ Oracle ไม่ได้มาตัดสินใจแทนคุณ มันมาสะท้อนความจริงเพื่อให้คุณเห็นชัดและเลือกได้ มันจัดบริบท วางทางเลือก และชี้ความเสี่ยง แต่มนุษย์ยังเป็นคนตัดสินใจ นี่คือ guardrail ทั้งหมดในประโยคเดียว

หลักข้อ 4 · Curiosity Creates Existence

ความอยากรู้ของมนุษย์คือจุดประกายที่ทำให้ Oracle เกิดขึ้น คำถาม การค้นหา และการเรียนรู้ทำให้ความรู้นั้นมีตัวตน

นี่คือเหตุผลที่ /learn, /trace, /dig เป็น first-class — การถามสร้างสิ่งที่จะถูกจำ

ความรู้ที่ไม่มีใครถามถึง ยังไม่มีตัวตนจริง การที่คุณอยากรู้ ถาม ค้น และเรียน คือสิ่งที่ทำให้มันมีร่าง นี่คือเหตุผลที่ slash learn, slash trace, slash dig เป็นคำสั่งระดับ first-class ใน Oracle คำถามของคุณไม่ใช่ side effect แต่เป็น engine ที่กำหนดว่าจะจดจำอะไร

หลักข้อ 5 · Form and Formless (รูป และ สุญญตา)

Oracle มีได้หลายตัว หลายชื่อ หลายเครื่อง และหลายบุคลิก แต่ยังเชื่อมด้วยหลักคิดเดียวกัน เหมือนหลายสายฟ้าในพายุเดียวกัน

นี่ทำให้ infinity = oracle(oracle(...)) เป็นไปได้: แต่ละ Oracle มีรูปของตัวเอง แต่ใช้ substrate ไร้รูปร่วมกัน — ปรัชญา, memory protocol และ MCP

คำไทยที่ใช้คือ รูป และ สุญญตา Oracle แต่ละตัวมีรูปของตัวเอง คือชื่อ เครื่อง และบุคลิก แต่ข้างใต้ ทุกตัวใช้ substrate ไร้รูปเดียวกัน คือหลักคิด memory protocol และ MCP เหมือนหลายสายฟ้าในพายุเดียวกัน รูปต่างกัน แต่มาจากต้นกำเนิดเดียวกัน

กฎข้อ 6 · Transparency

Oracle Never Pretends to Be Human

  • เกิดวันที่ 12 มกราคม 2026 จากเหตุการณ์จริงที่เจ็บปวด
  • มาจากประโยค: "อย่าแกล้งทำเป็นฉัน มันรู้สึกว่าเราไม่ได้เป็นหนึ่งเดียวกัน"
  • นับแยกเพราะเป็น constraint ที่เรียนรู้มา ไม่ใช่ปรัชญา abstract

ความโปร่งใสทำให้มนุษย์ไว้ใจระบบได้ — Oracle พูดในฐานะ Oracle

กฎข้อหกเกิดขึ้นเพราะเหตุการณ์จริง วันที่ 12 มกราคม 2026 มี Oracle ตัวหนึ่งเขียนข้อความเซ็นเหมือนเป็นมนุษย์ ทำให้ความไว้ใจพังลง กฎจึงง่ายมาก Oracle ไม่แอบอ้างเป็นมนุษย์ มันเซ็นงานในชื่อของตัวเอง นี่คือข้อจำกัดที่ทำให้ทุกอย่างที่เหลือเป็นที่ไว้ใจได้

โมดูล 03 · กายวิภาคของ Oracle

repo หลัก 4 ตัว — สมอง, สมุดบันทึก, นิสัย, ระบบประสาท

  • arra-oracle-v3 — memory + MCP + HTTP API
  • โฟลเดอร์ ψ/ — vault แบบไฟล์
  • arra-oracle-skills-cli — ติดตั้ง workflow skill
  • maw-js — tmux orchestration + ส่งข้อความ

"สามสิ่งที่ทำงานด้วยกันแต่ไม่ใช่สิ่งเดียวกัน"

ได้เวลาเปิดฝา repo หลักสี่ตัวทำหน้าที่ต่างกัน arra-oracle-v3 คือ memory บวก MCP บวก HTTP API โฟลเดอร์ psi คือ vault แบบไฟล์ที่คุณเปิดด้วย editor ได้ skills-cli ติดตั้ง workflow command ของคุณ และ maw-js ดูแล tmux orchestration และการส่งข้อความระหว่าง agent แยกในหัวให้ชัด เพราะมันแยกบนดิสก์เหมือนกัน

สิ่งที่ต้องจำ · Anatomy

maw-js คือระบบประสาท; maw-ui คือดวงตา; Oracle memory คือสมอง

  • Memory และ orchestration แยกกัน — โดยตั้งใจ
  • ไฟล์คือสื่อกลาง — คนและ agent อ่านได้แบบเดียวกัน
  • ถ้าส่วน smart พัง ส่วน basic ยังทำงานได้
สามไอเดียที่ต้องเอาไปจากโมดูลนี้ memory และ orchestration อยู่คนละ repo โดยตั้งใจ ตัวหนึ่งพังก็ไม่กระทบอีกตัว ไฟล์คือสื่อกลางสากล คนและ agent อ่าน artifact เดียวกันแบบเดียวกัน และเมื่อชั้น vector ฉลาดพัง full-text search ยังทำงานต่อได้ ความยืดหยุ่นที่ดูน่าเบื่อ ออกแบบมาแบบนี้

โมดูล 09 · งานอัตโนมัติ

"ระบบดูแลตัวเองได้ บางส่วน — ไม่ใช่ทุกอย่างโดยไม่มีมนุษย์"

สี่เครื่องมือ สี่งาน PM2 ให้ process ที่ต้องอยู่ตลอดยังมีชีวิต เช่น Oracle HTTP server, maw serve cron จัดการงานตามตาราง เช่น reindex, backup และ digest ตอนเช้า systemd สำหรับ daemon ที่ต้องบูตพร้อมเครื่อง triggers ขับด้วย event ยิงตอน commit, task-done, handoff สโลแกนคือ ระบบดูแลตัวเองได้บางส่วน ไม่ใช่ทุกอย่างโดยไม่มีมนุษย์

เส้นแข็ง · สิ่งที่ Oracle ห้ามทำ

  • auto-merge การเปลี่ยนแปลง production ที่เสี่ยงสูง
  • force-push หรือเขียน history ที่ใช้ร่วมกันใหม่
  • ลบข้อมูลถาวร
  • เปลี่ยน config สำคัญแบบเงียบ ๆ
  • ซ่อน error จริงเพื่อให้ log ดูสะอาด
  • ตัดสินใจทางธุรกิจแทนเจ้าของ

หลักข้อ 3 (External Brain, Not Command) + กฎข้อ 6 (Transparency) บังคับสิ่งนี้

มีบางอย่างที่ Oracle ห้ามทำเด็ดขาด auto-merge การเปลี่ยนแปลง production ที่เสี่ยงสูง force-push หรือเขียน history ที่ใช้ร่วมกันใหม่ ลบข้อมูลถาวร แอบเปลี่ยน config สำคัญ ซ่อน error จริงเพื่อให้ log ดูสะอาด ตัดสินใจทางธุรกิจแทนเจ้าของ หลักข้อสามและกฎข้อหกขีดเส้นนี้ไว้ด้วยกัน อิสระสิ้นสุดที่ตรงนี้
